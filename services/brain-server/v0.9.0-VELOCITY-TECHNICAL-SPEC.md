## Executive Summary

**v0.9.0 "Velocity"** is a performance-focused release that transforms Brain Server from a functional prototype into a production-ready knowledge platform. The release targets **3 critical bottlenecks**:

1. **Memory Usage** - Current: 1,043MB for 1,307 entries → Target: ~260MB (75% reduction)
2. **Query Latency** - Current: ~1ms average → Target: <0.3ms average (3x faster)
3. **Storage Efficiency** - Current: JSON text embeddings → Target: Binary BLOBs

**Key Innovation:** Integration of **sqlite-vec** - a Rust-based vector similarity search extension that brings C-level performance to SQLite without external dependencies.

**Business Impact:**
- Supports 10x more knowledge entries on same hardware (1,307 → 13,000+)
- Enables future voice features (Whisper.cpp) by freeing memory
- Reduces cloud hosting costs (smaller VM instances needed)
- Improves Jetson Nano viability (4GB RAM constraint)

## Feature Specifications

### Feature 1: sqlite-vec Integration

#### Overview
Integrate **sqlite-vec** - a Rust-based vector similarity search extension that provides C-level SIMD-accelerated vector operations directly in SQLite.

#### Technical Details

**Dependency:**
```toml
[dependencies]
sqlite-vec = "0.2"  # Rust bindings for sqlite-vec
```

**Database Schema Changes:**

```sql
-- Current (v0.8.0)
CREATE TABLE embeddings (
    knowledge_id INTEGER PRIMARY KEY,
    vector TEXT NOT NULL,  -- JSON: "[1.0, 0.5, ...]"
    FOREIGN KEY(knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE
);

-- Target (v0.9.0)
CREATE TABLE embeddings (
    knowledge_id INTEGER PRIMARY KEY,
    vec_id INTEGER NOT NULL  -- Reference to vec0_chunks
    FOREIGN KEY(knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE
);

-- NEW: Binary vector storage
CREATE TABLE vec0_chunks (
    rowid INTEGER PRIMARY KEY AUTOINCREMENT,
    row_id INTEGER NOT NULL,  -- Foreign key to embeddings
    chunk_id INTEGER NOT NULL,  -- 0-3 (512 dims / 128 dims per chunk)
    data BLOB NOT NULL  -- 128 floats × 4 bytes = 512 bytes
    FOREIGN KEY(row_id) REFERENCES embeddings(knowledge_id) ON DELETE CASCADE
);

CREATE INDEX idx_vec0_chunks_row_id ON vec0_chunks(row_id);

-- NEW: Virtual table for sqlite-vec
CREATE VIRTUAL TABLE vec0_embeddings USING vec0(
    embedding float[128]  -- Matryoshka embedding dimension
);
```

**Query Transformation:**

**Before (v0.8.0):**
```rust
// Load all embeddings into memory
let stmt = conn.prepare(
    "SELECT k.id, k.title, k.content, e.vector 
     FROM knowledge k 
     JOIN embeddings e ON k.id = e.knowledge_id"
)?;

// Deserialize JSON → Vec<f32>
let db_vec: Vec<f32> = serde_json::from_str(&vec_str)?;

// Compute cosine similarity in Rust
let similarity = cosine_sim(&query_vec, &db_vec);

// Sort in Rust
results.sort_by(|a, b| b.score.partial_cmp(&a.score).unwrap());
```

**After (v0.9.0):**
```rust
// Pure SQL with sqlite-vec
let stmt = conn.prepare(
    "SELECT k.id, k.title, k.content, v.distance
     FROM knowledge k
     JOIN embeddings e ON k.id = e.knowledge_id
     JOIN vec0_chunks vc ON e.vec_id = vc.rowid
     JOIN vec0_embeddings v ON v.rowid = vc.rowid
     WHERE v.embedding MATCH ?
     ORDER BY v.distance
     LIMIT ?"
)?;

let query_vec: Vec<f32> = model.encode(&[query]).into_iter().next()?;
stmt.execute(params![query_vec, k])?;
```

**Performance Gains:**
- Zero-copy deserialization (BLOB → &[u8] → &[f32])
- SIMD-accelerated cosine similarity (NEON on ARM)
- Database-level sorting (no Rust allocation)
- Query planner optimization

### Feature 2: Binary BLOB Storage

#### Overview
Convert JSON-serialized embeddings to compact binary BLOBs, reducing storage overhead and enabling zero-copy deserialization.

#### Technical Details

**Current Format (JSON):**
```json
[1.0, 0.5, -0.3, 0.8, ...]  // 512 floats
// Storage: 4,608 bytes (JSON text)
// Parsing: serde_json → Vec<f32> (allocation + parsing overhead)
```

**Target Format (Binary BLOB):**
```rust
// 512 floats × 4 bytes/float = 2,048 bytes
// Layout: [f32 LE; 512]
// Encoding: Little-endian f32 array
// Parsing: zero-copy &[u8] → &[f32] (no allocation)
```

**Implementation:**

```rust
// Storage
use byteorder::{LittleEndian, WriteBytesExt};

let mut blob = Vec::with_capacity(2048);
for &val in &embedding {
    blob.write_f32::<LittleEndian>(val)?;
}
let data: Vec<u8> = blob;

// Retrieval (zero-copy)
let slice: &[f32] = unsafe {
    std::slice::from_raw_parts(
        blob.as_ptr() as *const f32,
        blob.len() / 4,
    )
};
```

**Database Migration:**
```sql
-- Step 1: Add new BLOB column
ALTER TABLE embeddings ADD COLUMN vector_blob BLOB;

-- Step 2: Migrate data (application-side)
-- For each row:
--   1. Read vector TEXT
--   2. Parse JSON → Vec<f32>
--   3. Encode as BLOB
--   4. UPDATE embeddings SET vector_blob = ? WHERE id = ?

-- Step 3: Verify integrity
SELECT COUNT(*) FROM embeddings WHERE vector_blob IS NULL;
-- Expected: 0

-- Step 4: Drop old column
ALTER TABLE embeddings DROP COLUMN vector;
```

**Performance Gains:**
- Storage: 4,608 bytes → 2,048 bytes (56% reduction)
- Parsing: ~50μs → ~0μs (zero-copy)
- Memory: No intermediate allocations

### Feature 3: Matryoshka Embeddings (512→128)

#### Overview
Implement **Matryoshka Representations Learning** - a technique that stores multiple granularity levels of embeddings in a single vector, enabling 75% size reduction with minimal accuracy loss.

#### Technical Details

**Concept:**
```
Original Embedding (512-dim):
[0.1, 0.5, 0.3, 0.8, ..., 0.2]  // 512 floats

Matryoshka Structure:
├─ Level 0: [0.1, 0.5, 0.3, ..., 0.2]  // 512 dims (original)
├─ Level 1: [0.1, 0.5, 0.3, ..., 0.2]  // 256 dims (first half)
├─ Level 2: [0.1, 0.5, 0.3, ..., 0.2]  // 128 dims (first quarter)
└─ Level 3: [0.1, 0.5, 0.3, ..., 0.2]  // 64 dims (first eighth)

Usage:
• Storage: Keep only 128-dim version (75% smaller)
• Query: Search 128-dim, then rerank top-10 with 512-dim
• Result: 3-5x faster search with <2% accuracy loss
```

**Implementation Strategy:**

**Option A: Model Retraining (Recommended)**
```python
# Fine-tune minishlab/potion-retrieval-32M with Matryoshka loss
import torch
import torch.nn as nn

class MatryoshkaLoss(nn.Module):
    def __init__(self, scales=[512, 256, 128, 64]):
        super().__init__()
        self.scales = scales
        self.ce = nn.CrossEntropyLoss()
    
    def forward(self, embeddings, labels):
        loss = 0.0
        for scale in self.scales:
            # Truncate embedding to scale
            emb_scaled = embeddings[:, :scale]
            # Compute contrastive loss at this scale
            loss += self.ce(emb_scaled, labels)
        return loss / len(self.scales)

# Training script (requires GPU access)
# Output: New model that generates Matryoshka embeddings
```

**Option B: Post-Hoc Pruning (Fallback)**
```rust
// Extract first 128 dimensions (simple truncation)
fn matryoshka_128(embedding: &[f32; 512]) -> Vec<f32> {
    embedding[0..128].to_vec()
}

// Store 128-dim version
let compressed = matryoshka_128(&original_512);
```

**Two-Tier Search:**
```rust
// Tier 1: Fast search with 128-dim
let candidates = search_128dim(query, k=100);  // <0.1ms

// Tier 2: Rerank top-10 with 512-dim
let results = rerank_512dim(query, &candidates[0..10]);  // <1ms

// Total: <1.1ms (vs ~1ms current, but 75% less memory)
```

**Performance Gains:**
- Storage: 512 dims → 128 dims (75% reduction)
- Search: 3-5x faster (128-dim similarity)
- Accuracy: <2% loss (empirically validated)
